# AI 운영 프레임 — daangn_ad_reporter

이 앱을 "질문 답변기"가 아니라 **매장 지식을 축적하며 스스로 개선되는 운영 시스템**으로 만든다.
3계층: **위키(기억) → 워크플로우(순서) → 루프(완료 기준까지 반복)**. 결과는 다시 위키에 기록되어 다음 작업이 똑똑해진다.

핵심 원칙: 프롬프트에 경직된 미세 규칙을 쌓지 않는다. **매장별 사실은 위키에 두고, 프롬프트는 위키를 읽는다.**

---

## 1. LLM Wiki = 매장 위키 (지금 없는 핵심 계층)

지금 앱: SQLite에 **원자료**(projects / generated_content / performance_rows / report_content)만 있음.
빠진 것: **정리된 지식**. 그래서 매달 AI가 백지에서 다시 분석하고, 성과가 누적되지 않는다.

추가할 계층 = **매장 위키** (프로젝트=매장 단위로 1개, 마크다운 누적):

```
매장 위키 (store_wiki) 항목
- 시장 특성        : 지역 규모(소도시/광역시), 도달 인구 감, 계절/지역 이슈
- 추적 한계        : 이 업종/매장이 디지털로 안 잡는 전환 (예: 안경원 = 유입·전화·내방 추적 불가
                     → 문의 0은 정상. KPI는 쿠폰사용·전화·내방으로 본다)
- 검증된 자산      : 성과로 검증된 미끼/카피/타겟/소재 (예: "변색렌즈 0원" 후크 효율 좋음)
- 안 먹힌 것       : 돌려봤는데 효율 낮았던 것 (반복 방지)
- 과거 진단        : 지난 보고서의 핵심 병목·가설 (이번 분석이 이어받음)
- 사장님 사실/피드백: 행사 사실관계, 사장님이 정정한 내용 (모든 콘텐츠의 기준)
```

- 저장: `generated_content`에 `content_type="store_wiki"`로 매장당 1개 누적(또는 전용 테이블). 마크다운.
- 주입: 전략/소식글/세팅/보고서 **모든 프롬프트가 매장 위키를 컨텍스트로 먼저 읽는다.**
- 갱신: 보고서·성과 분석이 끝나면 "이번에 배운 것"을 위키에 **append/merge**(충돌은 표시).
- 효과: 안경원 추적 문제 같은 건 프롬프트 규칙이 아니라 **위키 사실 1줄**로 해결되고, 매달 누적된다.

## 2. Workflow = 입력 → 분석 → 생성 → 검수 → 저장 → 위키 반영

```
[입력]   매장 정보 + (선택)현재 광고 + (선택)리서치 + (선택)성과 데이터
  ↓
[위키]   매장 위키 로드 (없으면 입력으로 초기 생성)
  ↓
[분석]   전략 (위키 + 입력 기반)
  ↓
[생성]   소식글·제목·쿠폰 / 광고 세팅 / 제안서
  ↓
[검수]   루프(아래 3절): 완료 기준 통과까지 반복
  ↓
[저장]   DB + DOCX
  ↓
[반영]   이번 결과의 학습을 매장 위키에 기록 → 다음 작업이 더 똑똑해짐
```

성과 보고서도 같은 라인: 데이터 입력 → 위키 로드 → 진단 초안 → 수치/논리 검수 → 출력 → **위키에 이번 진단 기록**(다음 달이 이어받음).

---

## 3. Loop = 완료 기준 통과까지 반복 (이미 일부 있음: output_validator)

지금 `app/ai/output_validator.py`가 섹션 누락 1회 보정을 한다. 이걸 **출력별 명시적 완료 기준**으로 확장한다.

```
나쁜 종료: "괜찮게 만들어줘"
좋은 종료: 아래를 모두 만족할 때만 완료
```

| 산출물 | 완료 기준 (예시) |
|---|---|
| 전략 | 미끼 적합도 평가 / 시나리오 비교 / 권장+근거 / 변수통제 세팅 / 의사결정 기준 5개 모두 존재 |
| 보고서 | 시장 보정 1줄 / 병목 1구간 인과 / 업종 추적한계 반영 / 근거 기반 실행 / 판단기준 |
| 소식글 | 후크-혜택-신뢰-CTA-단서 구조 / 사장님 사실관계 일치 / 과장표현 0 |

루프 = 생성 → 기준 체크 → 미달 항목만 보정 → 재검 → 통과 시 종료(원본보다 나빠지면 원본 유지).

---

## 4. 내(AI)가 이 repo에서 일하는 방식 (운영 규칙)

매 작업:
1. `_ops/`의 이 문서 + `TASKS.md` + `ERROR_LOG.md` + 루트 `CLAUDE.md`를 먼저 읽는다.
2. `TASKS.md`의 최우선 TASK **1개만** 처리한다. (한 번에 여러 기능 갈아엎기 금지)
3. 현재 구조 파악 → 수정 → 실행/테스트 → 오류 시 원인 찾고 수정 루프 → 350 tests 통과 확인.
4. 완료 기준 충족 전엔 "완료"라고 말하지 않는다.
5. `TASKS.md` 상태 갱신 + 오류는 `ERROR_LOG.md`에 (증상/원인/해결) 기록.
6. 무엇을 왜 고쳤고 어떤 결과가 나왔는지 마지막에 보고.

---

## 5. 단계별 목표 (큰 틀 → 작은 TASK)

- **0 (완료)** 보고서 프롬프트 품질 — 시장 보정·인과 진단·업종 추적 반영. 실데이터 검증 완료.
- **1** 매장 위키 계층 신설 — 저장/로드/주입 배선 (1절). ← 다음
- **2** 전략/기획 프롬프트에 v4 프레임워크 주입(미끼평가·시나리오·변수통제·의사결정트리) + 위키 주입.
- **3** 위키 자동 갱신 — 보고서/성과 분석 후 학습을 위키에 반영(누적 루프 완성).
- **4** 완료 기준 루프 확장(3절) — 출력별 기준 체크.
- **5** (선택) 단계 축소/UI 단순화 — 위 품질이 받쳐진 뒤.

상세·우선순위·완료 기준은 `TASKS.md`.
